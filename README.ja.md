## AIQSE マニフェスト

私たちは、機械が実装を生み出し、品質がプロセスに作り込まれるという、ソフトウェア開発のより良い方法を明らかにしようとしている。この活動を通じて、私たちは以下の価値に至った。

- コードレビューよりも **品質の定義** を
- 実装の詳細よりも **実行可能な仕様** を
- 思い込みよりも **エビデンス** を
- 成果物の検査よりも **プロセスの管理** を
- 一度きりのテストよりも **継続的な検証** を

すなわち、前者に価値があることを認めながらも、私たちは後者により価値をおく。

## 三角形 (The Triangle)

AIQSE はパイプラインではなく、三角形の上に成り立ちます。

```text
                仕様 (Specification)
                何をしなければならないか
                 ╱         ╲
                ╱           ╲
               ╱  ( コード )  ╲
              ╱               ╲
     品質 (Quality) ────── 検証 (Verification)
  「正しい」とは何か         何が証明されたか
```

- **仕様 (Specification)** は期待される振る舞いを定義します。
- **品質 (Quality)** は「正しさ」を定義します — 入力処理から並行性、可観測性まで、9つの明示的な次元で。
- **検証 (Verification)** は両者が成立していることの証明を生み出します。

**コードは頂点ではありません。コードは三角形の内部です** — 生成され、使い捨てられ、再生成されるもの。実装が受け入れ可能なのは、三角形がその周りで閉じるとき、そのときだけです。どの頂点を誰が — 何が — 作ったかは問いません。三角形はエージェント非依存であり、それこそが完全自動化を矛盾ではなく設計目標にするものです。

## これは Spec-Driven Development の拡張ではない

SDD は線です: 仕様 → 実装。線が問えるのは「仕様どおりに作ったか?」だけであり、その検証は、検証対象と同じ仕様から、実装したのと同じエージェントによって導出されます。循環です。

三角形は、線に欠けているものを加えます: 「正しさ」の独立した定義 (品質) と、実装からではなく*そこから*導出される証明 (検証) です。

| | 形 | 正しさの定義 | 検証の出どころ |
|---|---|---|---|
| TDD | ループ | テストが表明したこと | 実装者自身のテスト |
| BDD | 線 | 期待される振る舞い | シナリオ (品質の一次元のみ) |
| SDD | 線 | 仕様に暗黙 | 仕様そのもの |
| **AIQSE** | **三角形** | **明示的・9次元** | **品質の頂点 — 実装から独立** |

## 産業品質の100年に学ぶ

ソフトウェアは産業革命の只中にあります: 実装は機械の仕事になりました。製造業は100年前にまったく同じ信頼の問題に直面し、それを解決しています — シューハート (1924): 品質は成果物ではなくプロセスの性質である。デミング: 事後の検査では遅すぎる。トヨタ: 品質が逸脱したら自ら止まる機械 (自働化)。AIQSE はこれらの教訓を直接輸入します。[docs/industrial-lessons.md](docs/industrial-lessons.md) を参照。

## 到達点は自律

三角形への人間の関与は設計ではなく、段階です。AIQSE は職人技から自律エンジニアリングまで、6つのレベルを定義します。

| レベル | 名称 | 人間の役割 |
|---|---|---|
| 0 | 職人技 (Craft) | すべてを書く |
| 1 | 支援付き (Assisted) | AI の提案を受けながら書く |
| 2 | 指示された生成 (Directed generation) | 仕様と品質を定義し、エビデンスを判定する |
| 3 | 監督された三角形 (Supervised triangle) | AI が起草した頂点を承認する |
| 4 | 限定された自律 (Bounded autonomy) | 基準を定め、例外を監査する |
| 5 | 自律エンジニアリング (Autonomous engineering) | 意図を表明し、システムを監査する |

統治原則: **自律は、検証が強くなる速さでしか進んではならない。** [docs/autonomy-levels.md](docs/autonomy-levels.md) を参照。

## 5つの原則

1. **品質はプロセスに作り込まれる。コードに検査で足すものではない。**
2. **三角形 — 仕様・品質・検証 — が第一級の成果物であり、コードはその内部である。**
3. **すべての頂点はエージェント非依存である。誰が作ろうと三角形は閉じなければならない。**
4. **検証は実装から独立していなければならない。**
5. **自律は、検証が強くなる速さでしか進んではならない。**

[PRINCIPLES.md](PRINCIPLES.md) を参照。
